# 1. 앱 라우터의 데이터 페칭(in App Router)
- App Router버전의 Next.js App에서는 `Server Component`로 인해 많은 변화가 발생 <br />
  → 그 중 하나가 `데이터 페칭`

### ※ Page Router 버전에서의 데이터 페칭
<img width="1285" height="345" alt="image" src="https://github.com/user-attachments/assets/9e65a6a9-9804-49b7-a812-3611cf8f3d31" />

- 사전 렌더링의 과정 중에 Next Server Side에서 또 다른 백엔드 서버나 데이터베이스로부터 데이터를 페칭 해오기 위해서
- 오직 서버 측에서만 실행이 되는 함수들을 이용

```tsx
// 사전 렌더링 과정 진행 중...
// 페이지 파일

// 1. SSR (서버 사이드 렌더링)
export async function getServerSideProps() {
  return { props: {...} }
}

// 2. SSG (정적 사이드 렌더링)
export async function getStaticProps() {
  return { props: {...} }
}

// 3. Dynamic SSG (동적 경로에 대한 정적 사이트 생성)
export async function getStaticPaths() {
  return { path: [...], fallback: ... }
}
```

- Page Router 버전에서는 Server Component라는 개념이 없었기 때문에(= 모든 컴포넌트가 Client Component로 동작) 페이지 컴포넌트 또한 Client Component로서 브라우저에서도 동일하게 한 번 더 실행
- Page Router 버전의 Next.js App에서는 페이지 컴포넌트 안에 데이터 페칭 로직을 직접 작성 <br />
  → 해당 로직은 서버 측에서만 실행되지 않음
- 브라우저에서도 Hydration 과정에서 한 번 더 실행이 되기 때문에 서버측에서만 데이터를 불러오기 위해서는 `getServerSideProps`나 `getStaticProps`등의 특수한 함수를 이용

<img width="607" height="425" alt="image" src="https://github.com/user-attachments/assets/ecd0a7e5-a38d-4aa1-9927-d835e6417b3e" />

- 서버 측에서 불러온 모든 데이터는 컴포넌트 트리의 최상단에 위치하는 페이지 컴포넌트에게만 props로 전달
- 최상단의 페이지 컴포넌트로부터 데이터를 필요로 하는 말단의 모든 컴포넌트들까지 props나 context API등 활용해서 힘들게 데이터를 전달하는 문제점
- App Router 버전부터는 해결 → Server Component 추가

### ※ App Router 버전에서의 데이터 페칭
- Server Component는 오직 서버에서만 실행

```tsx
// 서버 컴포넌트 (+ 비동기 함수) : 서버에서만 실행됨

export async function Page() {
  const data = await fetch('...');

  return <div>...</div>
}
```

- Client Component에는 Async 키워드를 사용할 수 없었음 <br />
  → 브라우저에서 동작시 문제를 일으킬 수 있기 때문에 권장X <br />
  → 예&#41; props의 전달, memoization, useMemo, useCallback 등 문제를 일으킬 수 있기 때문
- 기존의 getServerSideProps, getStaticProps 등의 함수 대체

<img width="457" height="229" alt="image" src="https://github.com/user-attachments/assets/d8c14367-95b9-430c-a87b-7d42d28ab8d2" />

- 컴포넌트 내부에서 자신이 필요로 하는 데이터를 불러와서 바로 렌더링에 사용 <br /> 
  → 데이터를 fetching하는 방식이 크게 변경 <br />
  → props, context API를 활용할 필요X

### ※ "Fetching data where it's needed"
- 데이터는 필요한 곳에서 직접 불러와라
